# Fases 10 e 11 — Relatórios/Histórico e Testes/Build/Deploy

Data: 2026-07-06

## Contexto

Implementação completa das Fases 10 e 11 do `PLANNING.md` do `guardpoint-manager` (painel Angular).

Backend (`http://localhost:8080/swagger/index.html`) já expõe `GET /turnos/historico` com filtros
`data_inicio`, `data_fim`, `usuario_id`, `posto_id`, `status`, `limit`, `offset`, mas:
- Não há endpoint de exportação (CSV/PDF) — precisa ser gerado no cliente.
- Não há endpoint de série temporal/tendência para turnos ou alertas — precisa ser agregado no
  cliente a partir dos dados já buscados.
- O schema de resposta do Swagger é genérico (`additionalProperties: true`), então o formato real
  (`{ data, total }` vs. array puro) precisa ser tratado defensivamente no service, seguindo o que
  já ocorre em `TurnosService` (`/turnos/ativos` retorna array puro) e `AlertasService` (`/alertas`
  retorna `{ data, total }`).

## Fase 10 — Relatórios e Histórico

### Módulo `features/relatorios/`

Rota `/relatorios`, guard `roleGuard(['admin', 'supervisor'])` (conforme seção 4 do PLANNING.md).

### `relatorios.service.ts`

- `listarHistorico(filtros, pagina): Observable<{ turnos: Turno[]; total: number }>`
  - Chama `GET /turnos/historico` com os filtros como query params.
  - DTO de resposta mapeado com a mesma convenção snake_case → camelCase de `TurnosService`
    (reaproveita `Turno` model; idealmente extrai o mapper de DTO para reuso, ou duplica localmente
    se a extração gerar acoplamento desnecessário entre features).
  - Trata resposta como `{ data: TurnoDto[], total: number }` OU array puro — detecção em runtime
    (`Array.isArray(resp) ? resp : resp.data`), com `total` derivado do array quando não vier
    explícito.
- Filtros de posto/usuário reaproveitam `PostosService.listar()` / `UsuariosService.listar()`
  já existentes (sem endpoint novo).

### `RelatoriosComponent`

- Formulário de filtro (Reactive Forms): `dataInicio`, `dataFim`, `postoId` (select), `usuarioId`
  (select), `status` (select), botão "Buscar" — mesmo padrão visual/estrutural de
  `turnos-list.component.ts`.
- Tabela `MatTable` + `MatPaginator` ligada a `limit`/`offset` do servidor (paginação real, não
  client-side). Usa `LoadingSpinner` e `EmptyState` compartilhados.
- Dois gráficos Chart.js, seguindo o padrão de `TurnosResumo` (`Chart.register(...registerables)`,
  `ChartConfiguration` tipado, destruição do chart anterior em `ngOnChanges`):
  - **Turnos por dia**: agregação client-side dos turnos do histórico filtrado, agrupados por
    `inicioPrevisto` (dia).
  - **Alertas por mês**: agregação client-side a partir de `AlertasService.listar()` (limit 100),
    agrupados por mês de `createdAt`.
  - Ambos os gráficos refletem apenas a janela de dados carregada — não há paginação "infinita" por
    trás deles, dado que o backend não expõe uma série temporal dedicada. Isso é aceito
    explicitamente (ver pergunta de brainstorming) e não precisa de aviso na UI além do que já é
    natural pelo filtro de datas aplicado.
- Exportação:
  - **CSV**: geração local de blob (`text/csv`), sem biblioteca nova, a partir das linhas
    atualmente carregadas na tabela.
  - **PDF**: `jspdf` + `jspdf-autotable` (novo pacote), importado via `import()` dinâmico dentro do
    componente de relatórios para não inflar o bundle inicial de outras rotas.

### Rotas e navegação

- Entry em `app.routes.ts` (`loadComponent`, lazy).
- Entry no sidebar (mesmo padrão dos itens existentes).

## Fase 11 — Testes, Build e Deploy

### Testes unitários (cobertura ampla, vitest via `@angular/build:unit-test`)

**Core:**
- `auth.service`, `auth.guard`, `role.guard`, `feature.guard`, `jwt.interceptor`,
  `tenant.service`, `api.service`, `notification.service`, `websocket.service`.

**Features (services):**
- `turnos.service`, `alertas.service`, `postos.service`, `usuarios.service`,
  `configuracoes.service`, `escalas.service`, `dashboard.service`, `relatorios.service` (novo).

**Componentes principais:**
- `login`, `dashboard`, `turnos-list`, `alertas-list`, `relatorios` (novo).
- Componentes compartilhados: `status-badge`, `empty-state`, `loading-spinner`.

Padrão de teste: `HttpTestingController` para mocks HTTP (já documentado em `AGENTS.md` seção 4.10).
Guards/interceptor cobrem cenários de autorização (permitido/negado por role) e refresh de token
(sucesso, falha, requisição concorrente).

### Dockerfile multi-stage

- **Stage 1** (`node:20-alpine`): `npm ci` + `ng build --configuration production`. Saída em
  `dist/guardpoint-manager/browser` (path padrão do builder `@angular/build:application`, sem
  `outputPath` customizado no `angular.json`).
- **Stage 2** (`nginx:alpine`): copia os arquivos estáticos da stage 1; `nginx.conf` como template
  processado por `envsubst` no entrypoint padrão da imagem `nginx:alpine`, permitindo configurar o
  host do backend (proxy_pass de `/api` e `/ws`) via variável de ambiente no runtime do container —
  necessário porque `environment.prod.ts` usa paths relativos (`/api/v1`, `/ws`).
- Gzip habilitado e cache de assets estáticos com hash (`outputHashing: all` já configurado no
  build de produção).

### CI/CD (`.github/workflows/ci.yml`)

- Disparo em push/PR para `main`.
- Steps: checkout → setup Node 20 → `npm ci` → `ng lint` → `ng test` → `ng build --configuration
  production`.
- Sem step de deploy — fica pronto para o usuário plugar deploy Railway quando tiver as
  credenciais (`RAILWAY_TOKEN`, etc.), conforme decisão de brainstorming.

### Otimizações

- Auditoria rápida por regressões: componentes sem `ChangeDetectionStrategy.OnPush`, imports não
  lazy que deveriam ser lazy.
- Ajuste dos budgets em `angular.json` se o bundle de produção ultrapassar os limites atuais
  (500kB warning / 1MB error) após adicionar `jspdf`/`jspdf-autotable` (mitigado pelo import
  dinâmico, mas budget pode precisar de ajuste fino).

## Decisões de brainstorming (registro)

1. **Export PDF**: `jsPDF` + `jspdf-autotable` (client-side), não impressão via navegador.
2. **Escopo de testes**: cobertura ampla (services + guards + componentes principais), não apenas
   lógica crítica nem smoke tests.
3. **CI/CD**: apenas lint + test + build; sem deploy automatizado ao Railway.
4. **Fonte dos gráficos de tendência**: agregação client-side a partir de `/turnos/historico` e
   `/alertas`, sem endpoint dedicado no backend.

## Fora de escopo

- Endpoint de exportação ou de série temporal no backend (fora do alcance deste módulo frontend).
- Deploy automatizado real no Railway (usuário ativa depois com suas credenciais).
- Testes end-to-end (Playwright/Cypress) — fora do que a Fase 11 do PLANNING.md pede
  (testes unitários + integração de componentes via TestBed).
